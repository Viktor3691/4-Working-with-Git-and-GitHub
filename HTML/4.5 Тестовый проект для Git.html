4.5 Тестовый проект для Git

Тестовый проект для Git

Для дальнейшей работы с Git нам понадобится некоторая папка test с тестовым проектом. Для примера сделаем в нем следующее содержимое:

  • index.html
  • styles.css

Сделайте папку для проекта и создайте в нем пустые файлы с теми именами, как показано выше в уроке.

Из предыдущих уроков вы уже знаете, что в Git есть специальное место, аналог папки резервного копирования. Это место называется репозиторий. В нем хранятся резервные копии кода проекта, а также настройки Git для этого проекта.

Этот репозиторий хранится в специальной папке .git внутри вашего проекта. Имя папки начинается с точки, и это не просто так. Дело в том, что под Linux, где изначально и был придуман Git, такая папка будет скрытая.

Итак, пусть у нас есть тестовый проект, описанный в предыдущем уроке. Как мы уже договорились, в нем будут следующие файлы:

  • index.html
  • styles.css

Теперь создадим репозиторий для этого проекта. Для этого в консоли перейдите в папку с проектом и выполните следующую команду:


git init

Инициализируйте Git репозиторий в вашем проекте.
Проверьте, что у вас появилась папка .git.

Отслеживание файлов в Git

В предыдущем уроке мы инициализировали Git в нашем проекте. И теперь мы можем с помощью команд Git делать сохранки проекта, или, по правильной терминологии, делать коммиты.
Однако, все не так просто. Дело в том, что в каждом проекте есть файлы, резервные копии которых не нужны. Например, файлы с кэшем. Поэтому в системе Git нужно явно указать, какие именно файлы мы хотим сохранять.
В Git файлы бывают двух типов: неотслеживаемые (untracked) и отслеживаемые (tracked).
Отслеживаемые файлы - это файлы, за изменениями которых Git следит. В случае изменения этих файлов, Git сообщит нам об этом.
Неотслеживаемые файлы - это файлы, за изменениями которых Git не следит. Изначально все файлы будут именно такими. А чтобы Git начал следить за файлами, нужно специально попросить его об этом. Давайте разберемся, как это сделать.
Для начала нам нужно научится смотреть состояния файлов. Для этого в Git есть специальная команда status:


git status

Пусть в вашей папке с проектом есть несколько файлов:

  • index.html
  • styles.css

Эти файлы изначально неотслеживаемые. В этом случае команда status выведет список этих файлов в таком виде:


Untracked files:
index.html
styles.css

Проверьте статус файлов в вашем проекте.

Сделайте в папке с проектом еще один файл. Проверьте статус. Убедитесь, что новый файл появился в списке неотслеживаемых.

Начало слежения за файлами в Git

Давайте теперь научимся включать слежение за файлами в Git. Для этого существует специальная команда add , после которой через пробел указывается имя файла:


git add index.html

После выполнения этой команды для конкретного файла, этот файл станет отслеживаемым. Проверим это с помощью команды status:


git status

В результате выполнения этой команды наш файл покажется уже в другом разделе:


Changes to be committed:
new file: index.html

Сделайте отслеживаемым каждый файл вашего проекта.

Начало слежения за папкой в Git

Добавлять вручную много файлов не очень удобно. Поэтому можно сделать отслеживаемыми все файлы заданной папки. Давайте посмотрим на практике. Пусть в нашем проекте есть следующая папка script:

  • index.html
  • styles.css
  • script/
    • index.js

Сделаем отслеживаем ее текущее содержимое:


git add script

При этом отслеживаемыми станут только те файлы, которые были в этой папке на момент выполнения команды, но не новые файлы.

Саму команду можно выполнять много раз. В этом случае новые файлы станут отслеживаемыми, а с теми, за которыми Git уже следит, ничего не произойдет.

Пусть в вашем проекте есть папка с файлами. Сделайте отслеживаемыми все файлы из этой папки.

Добавьте в эту папку новый файл. Проверьте, что новый файл будет неотслеживаемым.

Начало слежения за проектом в Git

Можно сделать отслеживаемыми все файлы папки с проектом. Для этого нужно выполнить следующую команду:


git add .

Сделайте отслеживаемым каждый файл вашего проекта.

Подготовка файлов к коммиту в Git

Слежение за файлами это только пол дела. Мы также должны указать Git, в какой момент запомнить состояние файла для последующего коммита. Это также делается с помощью многофункциональной команды add. Сейчас мы обсудим подробнее это тонкое место.

Пусть мы создали файл index.html. Изначально он будет не отслеживаемый:


Untracked files:
index.html

Сделаем его отслеживаемым:


git add index.html

Теперь он попадает в список для подготовленных к коммиту, как новый файл:


Changes to be committed:
new file: index.html

Изменим теперь текст файла. В этом случае файл файл попадет в раздел не подготовленных к коммиту, но уже как модифицированный:


Changes not staged for commit:
modified: index.html

Более того, этот файл будет сразу в двух разделах:


Changes to be committed:
new file: index.html

Changes not staged for commit:
modified: index.html

Как так? Дело в том, что Git сохраняет (индексирует) именно текущее состояние файла. Под состоянием понимается текущий текст файла, а также создан он или удален. Если же файл изменен, то просто так его изменения не сохранятся (не проиндексируются) в Git. Для этого нужно снова выполнить команду add. Фактически эту команду следует рассматривать как добавление текущего состояния файла к новому коммиту.

Именно поэтому мы видим наш файл в нескольких списках - это разные версии файла.

Давайте заставим Git проиндексировать текущее состояние нашего файла index.html, в который мы внести изменения. Выполним для него команду add:


git add index.html

После этого текущая версия станет готова к коммиту и попадет в соответствующий раздел:


Changes to be committed: modified: index.html

И так нужно делать каждый раз. Когда вы вносите изменения в файл, вы должны проиндексировать их для следующего коммита с помощью команды add. Конечно же, если вы хотите забирать эти изменения в следующий коммит.


Создайте новый файл. Проиндексируйте ваш файл. Посмотрите статус.
Внесите изменение в ваш файл. Посмотрите статус. Проиндексируйте ваш файл. Посмотрите статус.
Снова внесите изменение в ваш файл. Посмотрите статус. Проиндексируйте ваш файл. Посмотрите статус.

Выполнение коммитов в Git

После того, как мы проиндексировали файлы, мы можем выполнить коммит. Коммит сохраняет текущее состояния проекта. При этом к каждому коммиту мы должны добавить комментарий о сути внесенных изменений.

Коммиты выполняются с помощью команды commit. При этом после флага -m в кавычках мы должны написать комментарий к коммиту. Итак, давайте выполним наш коммит:


git commit -m “Комментарий к коммиту”

При желании, если комментарий большой, его можно набрать в текстовом редакторе. Для этого флаг -m не пишется и выполняется просто команда commit. После выполнения этой команды откроется привязанный вами текстовый редактор и вы сможете написать в нем комментарий к коммиту:


git commit

После этого коммит выполнен и вы можете посмотреть его в истории коммитов:


git log

Выполните коммит ваших проиндексированных файлов. Проверьте статус.

Внесите изменения в ваши файлы. Проверьте статус.
Проиндексируйте изменения. Проверьте статус.
Выполните коммит. Проверьте статус.

Состояния отслеживаемых файлов Git

Те состояния отслеживаемых файлов, которые мы видели в предыдущих уроках, имеют названия. Файл может принимать три состояния: modified (изменен), staged (индексирован) и committed (зафиксирован).

Измененным файл становиться когда в него были внесены правки, но их не сохранили. Индексированным файл становится после того, как его добавили для последующего включения в коммит. Зафиксированное состояние значит, что коммит уже сохранен и включен в ваш репозиторий.

В каком состоянии находится файл, если при проверке статуса нам выдало следующее:


Changes to be committed:
new file: index.html
файл находится в состоянии staged (индексирован).


Changes to be committed:
modified: index.html
Modified — файл изменён, но правки ещё не сохранены.

Changes not staged for commit:
modified: index.html
Файл index.html в указанном состоянии находится в состоянии modified (изменён)


nothing to commit, working tree clean
это означает, что в рабочем каталоге нет изменений, которые нужно зафиксировать (сделать коммит)
